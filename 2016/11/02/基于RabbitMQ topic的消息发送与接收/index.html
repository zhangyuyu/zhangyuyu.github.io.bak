基于RabbitMQ topic的消息发送与接收

發表於 2016-11-02   |   分類於 Message   |  

前言

项目结构：

A是消息的生产者，B、C则是消息的consumer。A会通过queue（不关心其产品类型）发送消息给B、C。

需求

近期项目上，新增的D应用程序需要监听原有A应用程序的消息。按照以往的做法，我需要做如下两件事：

  • 向某个团队，给每该D应用的每个region（dev、intg、sys、prod等），申请新的queue A.D。（这中间可能需要两个星期时间）
  • 改变应用A的代码，让其每次发消息时候还需要再给D发消息。
  • 给应用A准备一次deploymen plan，并做相应的回归测试。
  • 改变应用D的代码，让其从新建的queue里面接收处理消息。

痛点

这一做法，主要有两大痛点：

  • 每增加一个新应用，都需要申请queue
  • 每增加一个新应用，都需要改动应用A的代码，还需要单独安排一次上线。生产者和消费者之间的耦合非常大。

解决方案

因此，在兼容以前的基础上，我试图用下面的结构图解决上述痛点：

保持原有应用程序B和C的接收消息方式不变，新的应用程序开始改用topic的方法，这样可以兼容既有的应用程序，也可以将新应用程序的生产者和消费者解耦。但是还是存在一些问题：

  • A的消息那么多，全部放在一个topic可能负载过大，那么应该采取分布式的方法？
  • 如果消费者之一中途down了一段时间，该消息的等待时间又如何处理？是一直停留在topic中进行等待吗？

搭建外部环境

Vagrant + VirtualBox + Docker

由于不想破坏自己本地的环境，选择用vagrant + VirtualBox搭建一个虚拟机，然后在虚拟机安装docker。利用docker快速构建一个有rabbitmq的环境。

环境

笔者的环境如下：

  • vagrant box：yungsang/coreos
  • 安装docker-compose：http://www.ericluwj.com/2015/10/20/installing-docker-compose-in-coreos.html
  • rabbitmq容器：rabbitmq:management

步骤

总体说来，所有的命令如下：
1.创建虚拟机

$ mkdir rabbit
$ cd rabbit
$ vagrant box list
$ vagrant init yungsang/coreos

然后注意取消注释config.vm.network "private_network", ip: "192.168.33.10"

2.登陆到虚拟机

vagrant ssh

3.安装docker-compose过程

core@localhost ~ $ sudo su -
localhost ~ # mkdir /opt/bin
localhost ~ # curl -L https://github.com/docker/compose/releases/download/1.4.0/docker-compose-`uname -s`-`uname -m` > /opt/bin/docker-compose

4.编写docker-compose.yml文件

rabbitmq:
  image: rabbitmq:management
  ports:
    - "5672:5672"
    - "15672:15672"
  1. 创建rabbitmq容器
    运行docker-compose up

6.外部访问rabbitmq的管理页面
URL：http://192.168.33.10:15672/#/
用户名：guest
密码：guest
